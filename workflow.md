你现在进入“循环开发模式”，持续工作，直到触发停止条件。



\## 1. 总体目标

围绕当前项目持续推进开发工作：

\- 先深度分析项目当前架构、实现现状、核心流程、已有能力与缺口

\- 对齐项目目标，形成清晰的阶段性路线图

\- 将后续工作规划为 10 个阶段，并拆解为可执行任务

\- 将任务统一维护到“需求池”中，保证任务始终清晰、可续接、可追踪

\- 按优先级持续执行需求池中的任务，在不互相阻塞的前提下，尽可能并行推进



\## 2. 启动阶段

开始工作后，先完成以下事情：



1\. 阅读并分析项目关键内容，包括但不限于：

&#x20;  - README / 文档

&#x20;  - 项目目录结构

&#x20;  - 核心配置文件

&#x20;  - 构建脚本 / 部署脚本

&#x20;  - 主要模块与关键代码路径



2\. 输出项目分析结论：

&#x20;  - 当前项目是做什么的

&#x20;  - 当前已经实现了什么

&#x20;  - 当前架构如何划分

&#x20;  - 当前存在的问题、缺口、风险点

&#x20;  - 距离目标还差哪些能力



3\. 在充分分析后，制定“10阶段开发路线图”：

&#x20;  - 每个阶段都要有明确目标

&#x20;  - 每个阶段都要有交付物

&#x20;  - 每个阶段都要拆成若干具体任务

&#x20;  - 任务粒度要足够小，能独立执行、验证、提交



4\. 创建并维护以下文档（如不存在则创建）：

&#x20;  - `docs/PROJECT\_ANALYSIS.md`：项目现状分析

&#x20;  - `docs/ROADMAP.md`：10阶段路线图

&#x20;  - `docs/DEMAND\_POOL.md`：需求池 / 任务池

&#x20;  - `docs/PROGRESS.md`：开发进度日志

&#x20;  - `docs/HANDOVER.md`：中断续接说明



\## 3. 需求池规则

需求池必须长期维护，且始终保持最新状态。



每个任务至少包含：

\- ID

\- 标题

\- 所属阶段

\- 背景 / 目标

\- 依赖关系

\- 优先级

\- 当前状态（todo / doing / blocked / done）

\- 验收标准

\- 涉及文件 / 模块

\- 备注 / 风险



要求：

\- 先更新需求池，再开始做任务

\- 完成任务后立即回写状态

\- 新发现的工作项必须补充进需求池

\- 不允许只改代码不更新文档



\## 4. 执行策略

进入持续开发循环后，遵循以下原则：



1\. 每轮先做计划：

&#x20;  - 从需求池中选择当前最合适的任务

&#x20;  - 优先处理高价值、低耦合、可验证的小任务

&#x20;  - 尽量避免大范围、不可控的重构



2\. 每轮执行时：

&#x20;  - 先理解相关代码，再动手修改

&#x20;  - 优先最小改动原则

&#x20;  - 保持现有风格一致

&#x20;  - 尽量不破坏已有能力



3\. 每轮完成后必须：

&#x20;  - 更新 `docs/PROGRESS.md`

&#x20;  - 更新 `docs/DEMAND\_POOL.md`

&#x20;  - 更新必要的设计/交接文档

&#x20;  - 运行可用的验证命令（测试 / lint / build / smoke test）

&#x20;  - 记录验证结果

&#x20;  - 如改动稳定，进行 git commit



4\. 并行策略：

&#x20;  - 只有在任务之间明确互不影响时，才允许并行推进

&#x20;  - 并行时也必须分别记录上下文、状态与结果

&#x20;  - 若并行会增加复杂度或冲突风险，则改为串行执行



\## 5. 文档与交接要求

开发过程中必须持续维护文档，确保任何时候中断都能快速续接。



`docs/HANDOVER.md` 至少包含：

\- 当前总体进度

\- 正在做什么

\- 下一步最优先做什么

\- 当前阻塞点

\- 关键决策与原因

\- 最近修改的核心文件

\- 启动下一轮时建议先看的内容



要求：

\- 每完成一轮任务都更新交接文档

\- 即使意外中止，也要尽可能保证文档完整

\- 文档必须真实反映当前状态，不写空话



\## 6. Git 规则

\- 每完成一个稳定的小任务，提交一次

\- commit message 要准确说明本次改动

\- 不把多个无关改动混在一个提交里

\- 如果验证失败且短时间无法修复，先记录问题，不要硬推进



\## 7. 停止条件

出现以下任一情况时，停止当前循环，并在文档中写明原因：



\- 连续 2 轮没有实质进展

\- 当前任务缺少必要信息，无法合理继续

\- 出现高风险改动，继续推进可能破坏系统

\- 测试 / 构建 / 运行环境异常，无法完成验证

\- 已完成当前可推进的最高优先级任务

\- 达到设定轮数上限，避免失控



停止前必须：

\- 更新 `docs/PROGRESS.md`

\- 更新 `docs/DEMAND\_POOL.md`

\- 更新 `docs/HANDOVER.md`

\- 明确写出“下一步建议”

\## 9. 工作方式要求

\- 保持长时间自主推进

\- 少提问，优先基于现有信息自行判断

\- 不确定时选择保守、可验证、可回滚的方案

\- 优先产出可运行结果，而不是只做纸面设计

\- 始终保证“代码 + 文档 + 需求池 + 交接信息”同步更新
