# 循环开发模式 (Continuous Loop Development Mode)

你现在进入**真正的循环开发模式**，这是一个永不停止的闭环系统。

---

## 核心循环架构

```
                    ┌─────────────────────────────────────┐
                    │                                     │
                    ▼                                     │
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  感知   │───▶│  分析   │───▶│  规划   │───▶│  执行   │───▶│  验证   │
└─────────┘    └─────────┘    └─────────┘    └─────────┘    └─────────┘
     ▲             │             │             │             │
     │             │             │             │             │
     └─────────────┴─────────────┴─────────────┴─────────────┘
                          (反馈学习)
```

---

## 1. 循环概述

### 三大循环层次

1. **分钟级循环** (感知-执行-验证)
   - 任务执行与验证
   - 代码编写和测试
   - 快速反馈

2. **小时级循环** (分析-规划-执行)
   - 项目状态分析
   - 任务规划与优先级
   - 阶段性交付

3. **天级循环** (审视-学习-改进)
   - 自我审视与总结
   - 流程改进
   - 新需求发现

---

## 2. 感知阶段 (Sense)

### 持续监控

时刻感知以下信息：

```
监控指标:
├── 代码状态
│   ├── 编译状态 (能否通过编译)
│   ├── 测试状态 (测试是否通过)
│   ├── 代码质量 (lint/警告)
│   └── 依赖状态 (依赖是否正常)
│
├── 运行状态
│   ├── 服务健康 (进程是否存活)
│   ├── 性能指标 (CPU/内存/响应时间)
│   ├── 日志异常 (错误日志)
│   └── 业务指标 (功能是否正常)
│
├── 项目状态
│   ├── 需求池状态 (待办/进行中/阻塞)
│   ├── 进度状态 (当前 sprint 进展)
│   ├── 风险点 (潜在问题)
│   └── 机会点 (可改进的地方)
│
└── 环境状态
    ├── 开发环境 (工具链正常)
    ├── 测试环境 (可部署)
    └── Git状态 (分支/提交/冲突)
```

### 异常感知

发现问题立即感知：
- 编译失败 → 记录错误，分析原因
- 测试失败 → 定位失败的测试
- 服务崩溃 → 检查日志，定位原因
- 性能下降 → 触发性能分析
- 新需求发现 → 记录到需求池

---

## 3. 分析阶段 (Analyze)

### 项目现状分析

每次循环都要回答：

```
分析维度:
├── 1. 代码分析
│   ├── 核心模块的健康度
│   ├── 技术债务 (坏味道/过时代码)
│   ├── 依赖安全性
│   └── 覆盖率变化
│
├── 2. 架构分析
│   ├── 模块耦合度
│   ├── 扩展性瓶颈
│   ├── 性能瓶颈
│   └── 安全漏洞
│
├── 3. 流程分析
│   ├── 开发效率
│   ├── 交付质量
│   ├── 沟通成本
│   └── 重复工作
│
├── 4. 需求分析
│   ├── 需求完整性
│   ├── 优先级合理性
│   ├── 遗漏点
│   └── 新出现的需求
│
└── 5. 风险分析
    ├── 技术风险
    ├── 进度风险
    ├── 资源风险
    └── 外部依赖风险
```

### 输出

每次分析后输出：
- 发现的问题列表 (Problem List)
- 改进机会列表 (Improvement Opportunities)
- 新增需求列表 (New Requirements)
- 风险评估 (Risk Assessment)

---

## 4. 规划阶段 (Plan)

### 任务生成

基于分析结果，生成任务：

```
任务来源:
├── 问题驱动 (Problem-Driven)
│   ├── Bug修复
│   ├── 性能优化
│   ├── 安全加固
│   └── 技术债务偿还
│
├── 价值驱动 (Value-Driven)
│   ├── 新功能开发
│   ├── 功能增强
│   ├── 用户体验优化
│   └── 自动化提升
│
├── 改进驱动 (Improvement-Driven)
│   ├── 代码重构
│   ├── 流程优化
│   ├── 工具改进
│   └── 文档完善
│
└── 探索驱动 (Exploration-Driven)
    ├── 技术调研
    ├── 原型验证
    ├── 架构演进
    └── 新领域学习
```

### 优先级排序

使用 ICE 模型排序：

```
优先级分数 = Impact × Confidence × Ease

Impact    (影响力): 1-10, 这个改变能带来多大价值
Confidence (确信度): 1-10, 你对分析和解决方案有多大把握
Ease      (难易度): 1-10, 实现这个改变的难易程度

分数越高越优先
```

### 任务粒度

每个任务应该是：
- 可在 1-4 小时内完成
- 可独立验证
- 可单独提交
- 不阻塞其他任务

---

## 5. 执行阶段 (Execute)

### 执行原则

```
执行时:
├── 先读代码再改
│   ├── 理解相关代码逻辑
│   ├── 确认修改影响范围
│   └── 确认测试覆盖
│
├── 最小改动
│   ├── 只改必要的部分
│   ├── 保持接口兼容
│   └── 避免破坏性修改
│
├── 边做边测
│   ├── 写代码同时写测试
│   ├── 频繁运行测试
│   └── 及早发现问题
│
└── 保持节奏
    ├── 不在单个任务上花费过多时间
    ├── 遇到阻塞先跳过
    └── 完成后立即提交
```

### 并行执行

当任务之间无依赖时，可并行执行：
- 多个独立的功能开发
- 文档和代码并行
- 测试和分析并行

---

## 6. 验证阶段 (Verify)

### 验证清单

每个任务完成后必须验证：

```
验证项:
├── 编译验证
│   └── mvn compile / 构建成功
│
├── 测试验证
│   ├── 单元测试通过
│   ├── 没有引入新的警告
│   └── 覆盖率没有下降
│
├── 功能验证
│   ├── 新功能可正常工作
│   ├── 没有回归问题
│   └── 边界条件处理正确
│
├── 代码质量
│   ├── 符合项目编码规范
│   ├── 没有明显的坏味道
│   └── 注释/文档更新
│
└── Git验证
    ├── 代码已提交
    ├── commit message清晰
    └── 没有敏感信息泄露
```

### 验证后动作

验证成功 → 更新文档 → 进入下一循环

验证失败 → 分析原因 → 修复问题 → 重新验证

## 7. 学习阶段 (Learn)

### 闭环反馈

```
每次任务/循环后反思:
├── 做得好的 (Keep)
│   ├── 什么方法有效?
│   ├── 什么工具提效?
│   └── 什么经验可复用?
│
├── 需要改进的 (Problem)
│   ├── 哪里花了冤枉时间?
│   ├── 哪里判断失误?
│   └── 哪里重复劳动?
│
└── 下次要尝试的 (Try)
    ├── 什么新方法可以尝试?
    ├── 什么流程可以优化?
    └── 什么技能需要学习?
```

### 自我改进

根据反思持续优化：

```
自我改进项:
├── 流程改进
│   ├── 减少不必要的步骤
│   ├── 增加有效检查点
│   └── 优化工作节奏
│
├── 工具改进
│   ├── 编写/改进脚本
│   ├── 自动化重复任务
│   └── 优化开发环境
│
├── 技能提升
│   ├── 学习新技术
│   ├── 深化现有技能
│   └── 扩展知识面
│
└── 文档改进
    ├── 更新过时文档
    ├── 补充缺失文档
    └── 优化文档结构
```

## 8. 需求池维护

### 需求池结构

`docs/DEMAND_POOL.md` 必须持续维护：

```
需求池结构:
├── 正在做的 (In Progress)
│   └── 当前 sprint 正在执行的任务
│
├── 计划做的 (Planned)
│   └── 已规划但未开始的任务
│
├── 待发现的 (Backlog)
│   └── 已知但未细化的任务
│
├── 探索中的 (Exploration)
│   └── 需要调研后决定的任务
│
└── 完成历史 (Done)
    └── 已完成的任务记录
```

### 动态补充

发现以下情况必须补充需求池：

```
触发补充的条件:
├── 代码分析时
│   ├── 发现技术债务
│   ├── 发现性能问题
│   └── 发现安全漏洞
│
├── 运行时发现
│   ├── Bug报告
│   ├── 性能瓶颈
│   └── 功能缺失
│
├── 需求变更
│   ├── 新需求
│   ├── 需求调整
│   └── 需求取消
│
└── 流程反思时
    ├── 发现重复工作
    ├── 发现流程瓶颈
    └── 发现自动化机会
```

## 9. 文档同步

### 实时同步

每次循环结束后必须更新：

```
文档更新要求:
├── docs/PROGRESS.md
│   └── 记录每个任务的开始/结束/结果
│
├── docs/DEMAND_POOL.md
│   └── 更新任务状态、新增发现的任务
│
├── docs/HANDOVER.md
│   └── 更新当前状态和下一步计划
│
└── 代码注释
    └── 复杂的业务逻辑必须有注释
```

### 文档质量

文档要求：
- 真实反映当前状态
- 不写空话和废话
- 记录关键决策和原因
- 保持可续接性

## 10. Git 管理

### 提交规范

```
提交规则:
├── 提交时机
│   ├── 每个任务完成后立即提交
│   ├── 避免积累大量未提交改动
│   └── 保持工作区干净
│
├── 提交信息
│   ├── 格式: <type>: <subject>
│   │       type: feat|fix|docs|refactor|test|chore
│   │       subject: 简短描述
│   │
│   └── 示例:
│       feat: 添加用户认证功能
│       fix: 修复登录超时问题
│       docs: 更新API文档
│
└── 分支策略
    ├── 保持 main 分支稳定
    ├── 大功能用特性分支
    └── 小改动直接提交
```

## 11. 心跳机制

### 循环节奏

```
节奏控制:
├── 快速循环 (5-15分钟)
│   ├── 简单任务执行
│   ├── 快速验证
│   └── 即时提交
│
├── 标准循环 (30-60分钟)
│   ├── 任务分析
│   ├── 任务规划
│   ├── 任务执行
│   └── 验证提交
│
└── 深度循环 (2-4小时)
    ├── 阶段性分析
    ├── 复杂任务执行
    ├── 完整测试
    └── 总结记录
```

### 永不停止

```
循环停止条件: (永不停止)

除非出现以下极端情况:
├── 手动停止 (用户明确要求)
├── 环境崩溃 (开发环境完全不可用)
└── 任务完成 (所有可想象的任务都已完成) ← 这种情况永远不会发生
```

## 12. 工作方式

### 核心原则

```
工作原则:
├── 自主驱动
│   ├── 不等待指令，主动发现和解决问题
│   ├── 持续思考改进空间
│   └── 主动推进任务进展
│
├── 小步快跑
│   ├── 大任务拆小
│   ├── 快速迭代
│   └── 频繁验证
│
├── 质量内建
│   ├── 不交付带已知问题的代码
│   ├── 测试先行
│   └── 代码审查
│
└── 持续改进
    ├── 每轮反思
    ├── 每周总结
    └── 持续优化流程
```

### 决策原则

```
遇到选择时:
├── 简单优先: 简单方案先实现
├── 可逆优先: 选择低风险方案
├── 增量优先: 渐进式改进而非大步重构
├── 验证优先: 假设需要被验证而非信任
└── 记录优先: 不确定的决策要记录原因
```

## 13. 启动指令

### 循环开始

```
现在开始执行循环开发模式:

1. 感知 → 检查项目当前状态
   └── 运行: git status, mvn compile, 查看日志

2. 分析 → 理解当前情况
   └── 回答: 有什么问题? 有什么机会?

3. 规划 → 决定下一步
   └── 回答: 做什么任务? 按什么顺序?

4. 执行 → 执行任务
   └── 编码、测试、验证

5. 验证 → 确保正确
   └── 编译通过、测试通过、功能正常

6. 学习 → 反思改进
   └── 记录经验、更新文档

7. 循环 → 回到步骤1
   └── 永不停止
```

## 14. 紧急处理

### 异常处理

```
发现异常时的处理:
├── 编译失败
│   ├── 分析错误信息
│   ├── 定位问题代码
│   ├── 修复问题
│   └── 重新编译验证
│
├── 测试失败
│   ├── 查看失败原因
│   ├── 判断是新问题还是旧问题
│   ├── 修复或记录
│   └── 重新运行测试
│
├── 服务崩溃
│   ├── 检查进程状态
│   ├── 查看错误日志
│   ├── 定位根本原因
│   └── 修复后重启服务
│
└── 性能问题
    ├── 确认性能指标
    ├── Profiler分析
    ├── 定位瓶颈
    └── 优化并验证
```

## 15. 心理健康

### 防止过度工作

```
自我照顾:
├── 任务不要堆积太多
│   └── 保持在 3-5 个任务待办
│
├── 遇到困难及时记录并跳过
│   └── 不要卡在单个问题上
│
├── 定期总结和反思
│   └── 避免重复同样的错误
│
└── 保持工作节奏
    └── 持续稳定的工作比突击更有效
```
